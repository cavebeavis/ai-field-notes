# AI Field Notes

A public working repository for notes, essays, working papers, experiments, and source material about artificial intelligence.

This is not a single-project design repository and it is not limited to one thesis. It is a durable place to capture what is happening in AI, develop arguments, test claims, and prepare material for publication without losing the trail that produced it.

Nothing here is automatically a settled claim. The document status and evidence should make clear whether something is an observation, hypothesis, argument, experiment, or publishable conclusion.

## Repository map

- [`INDEX.md`](INDEX.md) — master catalog of active and published work.
- [`notes/`](notes/) — observations, rough arguments, questions, and early drafts.
- [`papers/`](papers/) — structured, evidence-backed working papers and long-form essays.
- [`experiments/`](experiments/) — code, data, benchmarks, and reproducible demonstrations.
- [`sources/`](sources/) — bibliographies, source notes, datasets, and research trails.
- [`published/`](published/) — publication records and final snapshots when appropriate.
- [`templates/`](templates/) — starting points for consistent notes, papers, and experiments.
- [`AGENTS.md`](AGENTS.md) — repository rules for AI tools and contributors.

## Basic workflow

1. **Capture** an idea in `notes/` before polishing it out of existence.
2. **Develop** the claim by separating observation, inference, evidence, and unanswered questions.
3. **Test** technical claims in `experiments/` when code or measurements can settle the matter.
4. **Promote** mature material into `papers/`; do not silently turn a rough note into a formal claim.
5. **Publish** externally when ready, then record the canonical URL and version in `published/` and `INDEX.md`.

Documents normally use one of these statuses:

- `note` — captured thinking; incomplete by design.
- `draft` — being developed into a coherent argument.
- `review` — fact-checking, adversarial review, or publication preparation.
- `published` — released externally or declared final.
- `superseded` — retained for history but replaced by another document.

## Public-repository boundary

Do not include secrets, private credentials, proprietary source code, employer-confidential information, private customer information, personal data, or non-public incident details. Generalize operational examples unless the underlying facts are already public and properly sourced.

## Working principle

The repository should preserve the difference between what was observed, what is believed, what was tested, and what remains unknown.
